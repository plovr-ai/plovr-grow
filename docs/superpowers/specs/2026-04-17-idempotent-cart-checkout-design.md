# Idempotent Cart Checkout — Design

**Issue**: [#278](https://github.com/plovr-ai/plovr-grow/issues/278)
**Date**: 2026-04-17
**Status**: Draft

## Problem

`POST /api/external/v1/carts/{cartId}/checkout` currently throws `CART_NOT_ACTIVE` (422) on the second call against an already-submitted cart. In the phone-AI scenario, network jitter and retries cause the same cart to be submitted multiple times. The second call should be idempotent and return the original order instead of failing.

Additionally, two concurrent checkout calls on the same cart currently both succeed and create duplicate orders — there is no concurrency guard. After this change, concurrent callers must converge on the same orderId.

`cancelOrder` is already idempotent via atomic CAS. `checkout` should follow the same pattern.

## Goals

1. Second checkout on an already-submitted cart returns the original order with HTTP 200.
2. First checkout still returns HTTP 201.
3. Concurrent checkouts on the same cart produce **exactly one** order; all callers see the same orderId.
4. If order creation fails, the cart stays reusable (not stuck in `submitted`).
5. Semantics mirror `cancelOrder`: second call ignores any differing request body (true idempotency).

## Non-Goals

- No cross-cart idempotency key header (phone-AI retries use the same cartId).
- No `Order.cartId` reverse link.
- No retroactive cleanup for historical `submitted` carts with `order_id = NULL` (they remain gated by the legacy `CART_NOT_ACTIVE` path; phone-AI does not retry them).

## Design Decisions

Three questions drove the design, all resolved to the most user-friendly option:

| Question | Chosen | Rationale |
|---|---|---|
| Second request arrives before first wrote `orderId` | Short retry loop (5 × 100ms) on server | Transparent to client; matches phone-AI real-time expectations |
| Second request body differs from first | Ignore new body, return original order | Matches `cancelOrder` (ignores new `reason`); avoids fragile body-diff checks |
| Order creation fails after CAS claim | Rollback cart to `active` | Cart stays reusable; matches "cart is a mutable container" mental model |

## Data Model

Add `orderId` (nullable) to `Cart`. No `UNIQUE` constraint — rollback + retry can legitimately overwrite a stale value.

```prisma
model Cart {
  id           String     @id
  tenantId     String     @map("tenant_id")
  merchantId   String     @map("merchant_id")
  status       String     @default("active")
  salesChannel String     @map("sales_channel")
  notes        String?    @db.Text
  orderId      String?    @map("order_id")
  deleted      Boolean    @default(false)
  createdAt    DateTime   @default(now()) @map("created_at")
  updatedAt    DateTime   @updatedAt @map("updated_at")
  tenant       Tenant     @relation(fields: [tenantId], references: [id])
  merchant     Merchant   @relation(fields: [merchantId], references: [id])
  order        Order?     @relation(fields: [orderId], references: [id], onDelete: SetNull)
  cartItems    CartItem[]

  @@index([tenantId])
  @@index([tenantId, status])
  @@index([merchantId])
  @@index([orderId])
  @@map("carts")
}
```

`Order` gets a reverse relation: `carts Cart[]`.

**Migration**: Generated by `prisma migrate`. Expected SQL:

  ADD CONSTRAINT `carts_order_id_fkey`

## Concurrency Strategy — CAS Claim + Rollback

Three candidates were considered:

1. **CAS claim first, create order, rollback on failure** (chosen) — matches the codebase's existing `cancelOrder` CAS pattern. No nested transactions. No orphan orders.
2. **`SELECT ... FOR UPDATE` inside a transaction** — rejected. `createMerchantOrderAtomic` already owns its own `$transaction` with P2034 retry logic; nesting a longer-held row lock around it would serialize checkout for ~300ms per concurrent request.
3. **Let both create, rely on `Cart.orderId UNIQUE` to pick a winner, cancel loser's order** — rejected. Produces orphan orders that need to be canceled against the POS — dirty side effects unacceptable in phone-AI contexts.

### Flow

```
1. getCart → resolveExistingOrder(...)
   ├─ status=submitted + orderId    → return { ...order, alreadyExists: true }     // HTTP 200
   ├─ status=submitted + orderId=Ø  → waitForConcurrentCheckout(...)               // 5×100ms poll
   ├─ status=cancelled              → throw CART_NOT_ACTIVE
   ├─ status=active  + items=0      → throw CART_EMPTY
   └─ status=active                 → fall through to step 2
2. claimCartForCheckout: CAS updateMany({ id, tenantId, status:"active" } → { status:"submitted" })
   ├─ count=0  → lost race → waitForConcurrentCheckout(...)
   └─ count=1  → continue
3. orderService.createMerchantOrderAtomic(...)
   ├─ throws  → rollbackCheckoutClaim (CAS back to active when orderId still null), rethrow
   └─ succeeds → continue
4. attachOrderId: update cart.orderId = order.id
5. return { orderId, orderNumber, alreadyExists: false }                           // HTTP 201
```

### waitForConcurrentCheckout

- Poll cart 5 times at 100ms intervals.
- See `submitted + orderId` → fetch order, return with `alreadyExists: true`.
- See `active` (peer rolled back) → recursively re-enter `checkout` once (bounded recursion depth of 1 via a private entry point).
- Still `submitted + orderId=NULL` after 5 rounds → throw `CART_CHECKOUT_IN_PROGRESS` (HTTP 409). Peer is stuck; caller should diagnose.

### rollbackCheckoutClaim

```ts
updateMany({
  where: { id, tenantId, status: "submitted", orderId: null, deleted: false },
  data: { status: "active" }
})
```

The `orderId: null` predicate guarantees we never clobber a cart that has already been successfully bound to an order.

## API Contract

No change to request schema. Response body unchanged:
```json
{ "success": true, "data": { "orderId": "...", "orderNumber": "..." } }
```

HTTP status distinguishes the two cases:
- **201 Created** — this call created the order
- **200 OK** — this call returned a previously created order (idempotent hit)

The `alreadyExists: boolean` field exists only on the service return type (`CheckoutResult`); the route handler consumes it to pick the status code and does not forward it.

## Testing

### Unit (src/services/cart/__tests__/cart.service.test.ts — extend)

1. submitted + orderId → idempotent return, no order creation call
2. submitted + orderId=null → retry sees orderId → returns existing order
3. submitted + orderId=null → retry times out → throws `CART_CHECKOUT_IN_PROGRESS`
4. cancelled → `CART_NOT_ACTIVE` (preserve existing)
5. active + items=0 → `CART_EMPTY` (preserve existing)
6. CAS loses → peer succeeded → returns peer's order
7. CAS loses → peer rolled back → re-enter creates new order
8. order creation throws → rollback called, cart is `active`, error rethrown
9. second call with different body → still returns original order (true idempotency)

### Cleanup Sanity

Existing `cleanupTestData` deletes `cart*` before `order*`. The new FK is `cart.order_id → orders.id` with `ON DELETE SET NULL`, so the existing deletion order is safe. Does not trip MISTAKES.md #128 (cart is the "child" relative to the new FK, and it's already deleted first).

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Rollback CAS fails silently → cart stuck `submitted + orderId=null` | Rollback is fire-and-forget logged; next caller's `waitForConcurrentCheckout` will time out with `CART_CHECKOUT_IN_PROGRESS`. Acceptable — indicates upstream DB issue. |
| Second request fires WHILE first is in its 300ms order-creation window → second does 5×100ms = 500ms wait | Acceptable latency for phone-AI retries (which typically fire seconds apart, not hundreds of ms). |
| Stale `submitted` carts from before this migration (orderId=NULL) | Legacy behavior preserved: they throw `CART_NOT_ACTIVE`. Phone-AI never retries them. |
| `cartRepository.attachOrderId` after successful order creation fails (DB glitch) | Order exists but cart is `submitted + orderId=NULL`. Next retry polls, times out, returns `CART_CHECKOUT_IN_PROGRESS`. Rare (post-success failure) — acceptable for first iteration; can add background reconciliation later. |
| MISTAKES.md #163 / #170 (migration uses wrong VARCHAR or table name) | Rely on `prisma migrate dev` to generate SQL. Do not hand-write. |

## Out of Scope

- Client-provided `Idempotency-Key` header (future work).
- Auto-reconciliation background job for `submitted + orderId=NULL` carts.
- Adding `UNIQUE` on `Cart.orderId` (rollback can legitimately overwrite).
- `Order.cartId` field.
